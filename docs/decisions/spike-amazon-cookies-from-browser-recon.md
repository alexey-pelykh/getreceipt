## Question

Does getreceipt collect Amazon order history end-to-end via an imported browser session, and — since #228 flagged amazon.de as client-rendered — is the shared `/gp/css/order-history` HTML-scrape still valid for amazon.fr / amazon.com?

## TL;DR

- **The cookies-from-browser import works perfectly.** A complete, signed-in Amazon cookie set was lifted from Chrome for **both** marketplaces (amazon.fr: 14 cookies incl. `at-acbfr`/`sess-at-acbfr`/`x-acbfr`; amazon.com: 17 incl. `at-main`/`sess-at-main`/`x-main`). `[VALIDATED]`
- **#240 is a STALE-MARKER bug, not an SPA migration.** The live `/gp/css/order-history` returns **HTTP 200 "Vos commandes"** (~930 KB, signed-in) whose order list is **server-rendered**: **10 `order-card` divs** per page, each carrying its real order id in `data-csa-c-slot-id="amzn1.yourorders.order-card.<ORDER_ID>"`, a `<span class="num-orders">67 commandes</span>` total, and **server-side `startIndex` pagination** (7 pages, `?startIndex=0…60`). The order ids in that server HTML are byte-for-byte the same ones a raw browser-cookie `curl` returns. What actually changed are the adapter's **markers**: the legacy `ordersMarker` (`id="ordersContainer"`) is gone (the container is now `your-orders-content`), and the list page no longer emits `orderID=` invoice anchors — the order ids moved to the `order-card` slot-id attribute. `[VALIDATED]`
- Because `isOrderHistoryPage()` keys on the absent `id="ordersContainer"`, the adapter **throws a false `reauth_required` on a valid, order-bearing 200 page** (`adapter.ts:292-294`) — masking a marker migration as an auth failure. `[VALIDATED]`
- **The per-order *detail* on the list card is client-decrypted** (10 `csd-encrypted-sensitive` blocks, decrypted in-browser by Amazon's Siege CSD library — `SiegeClientSideDecryption.decryptInElement`, keyed on the `csd-key` cookie). But getreceipt does **not** read detail off the list card: its model (`wire.ts`) is to take each order id and fetch the **per-order invoice print page** (`/gp/css/summary/print.html?orderID=<ID>`) for items/prices. So the list-card encryption does not, by itself, block getreceipt. `[VALIDATED]` (encryption present) / `[INFERRED]` (irrelevant to getreceipt's path)
- **Intermittently, the same request instead gets `302 → /ap/signin?...max_auth_age=0` (the `amzn_retail_yourorders` realm)** — a step-up / freshness re-auth on a session whose identity cookies are valid. Observed on the first requests to both `.fr` and `.com`, then a 200 on the next request → account/context-gated (#185). `[VALIDATED]` (behaviour) / `[INFERRED]` (A-B mechanism)
- **Net (the maintainer's amazon.fr account): order *enumeration* IS collectable from the server HTML** — read order ids from the `order-card` slot-ids (not `orderID=` anchors), follow `startIndex` pagination, **no headless render needed.** (amazon.de renders the **same** server-rendered cards for a populated window — a raw `curl` of `?timeFilter=year-2025` returned 3 `order-card` divs; #228's "SPA" was an **empty default time window** misread as card-absence, not a separate arm — see § Single server-rendered model.) The adapter fails only because three string constants (`ordersMarker`, the `orderID=`-anchor extraction, and likely `nextPageClass`) are stale. The last piece — whether the per-order invoice print page still renders server-side (it decides end-to-end `collect()`) — was **validated server-rendered on 2026-06-25** (the sibling HQ collect-leg recon fetched `print.html?orderID=` → 200, a 191 KB real invoice) and **not re-tested this session**, so end-to-end collection is very likely intact behind the selector fix. Not an auth defect — a stale-marker/selector migration (#240) plus an intermittent step-up (#185).

## Correction (what the first draft got wrong, and why)

The initial spike concluded: *"no server-side `orderID=` invoice anchors… the whole order-history surface is now client-rendered… the orders are genuinely fetched by a later client request… getreceipt cannot currently collect amazon.fr order history… do not build another marker-scrape."*

**All of that was wrong.** The 937 KB body the spike itself captured contains the server-rendered order list:

- `class="order-card js-order-card"` → **10** occurrences; `data-csa-c-slot-id="amzn1.yourorders.order-card.402-2434601-9115567"` (and 9 more real order ids).
- `<span class="num-orders">67 commandes</span>`, `a-pagination`, `?startIndex=0…60` (7 pages).

**Root cause of the error:** the characterization probe (`spa-characterize.mjs`) only tested for the adapter's own marker set — `id="ordersContainer"`, `ABYourOrders`, hydration-JSON, XHR path hints — and **never canaried a known-present order id**. Absence-of-those-markers was mis-read as absence-of-orders (an "absence is only evidence if the query actually ran" failure), then compounded into a single-dimension leap ("my marker is gone → the whole page must be a client-rendered SPA") — **anchored on the #228 amazon.de ABYourOrders SPA finding**, which this account's page superficially resembled (same missing `id="ordersContainer"`). A one-line `grep '402-'` against the captured body would have falsified the conclusion immediately. (I then *over-corrected the other way* — treating #228's amazon.de result as a genuine card-less "A/B arm"; a populated-window `.de` `curl` falsified that too. amazon.de renders cards server-side; #228 had queried an empty default window. See § Single server-rendered model.) The narrow literal claim "no `orderID=` anchors" was true; every inference drawn from it was not.

## What actually happens (mechanism)

The adapter GETs `/gp/css/order-history` over the Chrome-impersonating transport with the imported cookie jar and `redirect: 'manual'` (`requestSession`, `adapter.ts:334`), then requires `id="ordersContainer"` to consider the body a real order page (`listAllOrders` → `isOrderHistoryPage`, `adapter.ts:290-295`). Two live responses were observed for that exact request:

| Response | What the adapter does | Reality |
|---|---|---|
| **`302 → /ap/signin?openid.pape.max_auth_age=0&…assoc_handle=amzn_retail_yourorders_fr`** | `requestSession` sees `/ap/signin` in `location` → `reauth_required` (`adapter.ts:356-358`) | A **step-up** for the your-orders realm; identity cookies are valid, Amazon wants a *fresh* auth for this sensitive page (#185) |
| **`200` "Vos commandes", server-rendered order list, no `id="ordersContainer"`, no `orderID=` anchors** | `isOrderHistoryPage()` → false (marker absent) → `reauth_required` (`adapter.ts:292-294`) | The **real order page, with 10 server-rendered order cards** (order ids in `data-csa-c-slot-id`). The adapter mis-labels a working, order-bearing page as an auth failure because its marker is stale — the orders are right there |

The verdict string is identical for both (`browserSessionReauthRequired(CANONICAL_DOMAIN)`), so a caller cannot tell a genuine step-up from a stale-marker mismatch — and `CANONICAL_DOMAIN` means an **amazon.fr** run reports *"Re-authentication required for amazon.com"* (cosmetic, but confusing). `[VALIDATED]`

### The order-history page, characterized (for the fix — #230/#240)

The 200 response (`https://www.amazon.fr/gp/css/order-history`, 936 KB):

- Signed-in: `<title>Vos commandes</title>`, `nav-line-1` account nav present; only 2 `ap/signin` refs (nav sign-out), **no `ap_email`/`ap_password` login form** → not a login page.
- **Server-rendered order list** (NOT an empty SPA shell): container is **`your-orders-content`** (not the legacy `id="ordersContainer"`); each order is a `<div class="order-card js-order-card" … data-csa-c-slot-id="amzn1.yourorders.order-card.<ORDER_ID>">`. **10 cards** on page 1; `<span class="num-orders">67 commandes</span>`; server-side pagination `/your-orders/orders?startIndex=0…60` across **7 pages** (10/page). The order ids extracted from the slot-ids match a raw browser-cookie `curl` exactly. `[VALIDATED]`
- **Order-id extraction has moved**: the list page no longer emits `/gp/css/summary/print.html?orderID=…` invoice anchors, so the adapter's `INVOICE_ANCHOR_RE` finds nothing. The order ids now live in the `order-card` `data-csa-c-slot-id` attribute — the fix reads them there, then constructs the invoice URL from `ENDPOINTS.invoicePrint` + the id (the adapter already knows that URL shape). `[VALIDATED]`
- **Per-order *detail* on the card is Siege-CSD-encrypted**: each card wraps a `csd-encrypted-sensitive` block decrypted in-browser (`SiegeClientSideDecryption.decryptInElement`, `csd-key` cookie). This gates the items/prices *as rendered on the list card* — but getreceipt reads detail from the **invoice print page**, not the card, so this is not on getreceipt's collection path. `[VALIDATED]` (encryption) / `[INFERRED]` (off-path for getreceipt)
- Same shape across routes: the legacy `/gp/css/order-history`, the modern `/your-orders/orders`, and `/your-orders/orders?timeFilter=year-2025` all return the same server-rendered `order-card` list (no `id="ordersContainer"`, `your-orders-content` present). So the adapter is pointed at a fine endpoint — it just reads stale tokens. `[VALIDATED]`

**Fix direction** (small, server-side, no headless):

1. **`ordersMarker`** (`LISTING.ordersMarker`, `wire.ts:77`): replace `id="ordersContainer"` with a token the current page reliably carries — `your-orders-content`, or the presence of `class="order-card"` / the `num-orders` span. (Don't hard-code a single legacy token — see the amazon.de note below.)
2. **Order-ref extraction** (`INVOICE_ANCHOR_RE`, `wire.ts:221`): read order ids from `data-csa-c-slot-id="amzn1.yourorders.order-card.<ID>"`, then build the invoice URL from `ENDPOINTS.invoicePrint` + `ORDER_QUERY.orderId` (the adapter already models this). The list-card order date is inside the CSD blob, so the date basis likely has to come from the invoice page too — verify.
3. **Pagination** (`nextPageClass: 'a-last'`, `wire.ts:79`): the captured page uses `a-pagination` with `startIndex` offsets; confirm whether `a-last`/`a-disabled` still marks the final page or whether pagination should be driven by `startIndex` up to `num-orders`.

**The remaining question for end-to-end `collect()`:** does `/gp/css/summary/print.html?orderID=<ID>` still render the invoice **server-side**? getreceipt's detail parse depends on it. It was **validated server-rendered on 2026-06-25** (the sibling HQ collect-leg recon: `print.html?orderID=` → 200, a 191 KB real French invoice → renders to PDF) and was **not re-tested this session** — so `collect()` very likely needs only the three selector fixes above and never touches Siege CSD. Worth a quick re-confirm given Amazon's DOM churn (the very churn that staled the list markers), but this is a re-validation, not an open unknown. `[VALIDATED 2026-06-25]` / `[OPEN: re-confirm]`

### Single server-rendered model, gated by `timeFilter` (the amazon.de reconciliation, corrected)

> **Correction round 2 (2026-07-02).** After the round‑1 fix above, an earlier version of *this* section (commit `8cca89e`) reconciled amazon.fr (server-rendered) vs amazon.de (#228's "ABYourOrders skeleton SPA") as **two coexisting A/B arms**, one of which "genuinely needs headless/XHR." A raw browser-cookie `curl` of amazon.de **falsifies that too**, so the section is rewritten below.

- **amazon.de renders order cards server-side, exactly like amazon.fr — when the window has orders.** `curl 'https://www.amazon.de/your-orders/orders?timeFilter=year-2025'` (browser cookie jar) returns **3 server-rendered** `<div class="order-card js-order-card" … data-csa-c-slot-id="amzn1.yourorders.order-card.<ID>">`, `<span class="num-orders">3 orders</span>`, the `AUIClients/YourOrdersOrdersView` bundle (**not** `ABYourOrdersWebAppAssets`), and the same Siege-CSD detail blocks. Structurally identical to the amazon.fr capture. `[VALIDATED]`
- **What #228 actually saw was an empty *window*, not a card-less *page*.** #228 requested `/gp/css/order-history` with **no `timeFilter`** — the default "past three months" — on a **low-activity** account (3 orders in *all* of 2025). That window was almost certainly **empty**, so Amazon returned an empty/skeleton shell (the `ABYourOrders` progressive-enhancement placeholder) with no cards, which #228 read as "cards positively absent → client-rendered SPA → needs headless/XHR." Same **absence-dual** error as this spike's round‑1 amazon.fr mistake — *an empty result is not a page that can't render* — with a different proximate cause (empty **window** on .de vs stale **markers** on a populated .fr page). `[INFERRED]` (empty-window cause — #228's raw body was deleted, so not byte-canaried) / `[VALIDATED]` (.de renders cards for a populated window)
- **Corrected model — one shape, `timeFilter`-gated.** Both marketplaces expose a **single server-rendered enumeration model**: `order-card` slot-ids paginated by `<select name="timeFilter">` (`last30`, `months-3` = default, `year-YYYY` 2009→now) and `startIndex`. An **empty window renders an empty/skeleton shell** and must **not** be read as "SPA / cannot collect." There is **no evidence of a separate, permanently-uncollectable client-rendered arm.** The #240 fix needs **no** headless render, **no** XHR-extraction engine, and **no** shape-detection branch.

My compounded error: round 1 read a *populated* page as an SPA (missed the cards); round 2 then read #228's *empty-window* result as proof of a genuine SPA "arm." Both are the same failure — treating an absent/empty result as evidence the page can't render server-side. The unifying guard is to **query a window that has orders and canary a known id** before any "shape" verdict. `[VALIDATED]` (fr shape, this spike) / `[VALIDATED]` (de shape, populated-window curl)
